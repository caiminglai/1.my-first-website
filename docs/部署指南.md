## 1. 项目结构模板

所有同类项目建议统一使用以下目录结构：

```
项目根目录/
├── app/                    # React 前端应用
│   ├── src/
│   │   ├── api/           # API 客户端封装
│   │   ├── components/    # 通用组件
│   │   ├── hooks/         # 自定义 Hooks
│   │   ├── pages/         # 页面组件
│   │   ├── sections/      # 页面区块（大组件）
│   │   ├── types/         # TypeScript 类型定义
│   │   └── data/          # 静态数据（少量）
│   ├── public/            # 静态资源
│   ├── vite.config.ts     # Vite 配置
│   ├── tsconfig.json      # TypeScript 配置
│   ├── tailwind.config.js # Tailwind 配置
│   └── package.json
│
├── backend/               # Express.js 后端
│   ├── db/               # 数据库层
│   │   ├── index.js       # 数据库入口
│   │   └── schema.sql     # 建表 SQL
│   ├── middleware/       # 中间件（安全、限流、日志）
│   ├── routes/           # 路由定义
│   ├── services/         # 业务逻辑层
│   ├── scripts/          # 工具脚本（备份、迁移等）
│   ├── utils/            # 工具函数
│   ├── public/           # 管理后台静态页面（可选）
│   ├── .env              # 环境变量（不提交 Git）
│   └── server.js         # 入口文件
│
├── data/                 # 数据库文件（不提交 Git）
│   ├── tongwuyiming.db   # 主数据库

│
├── docs/                 # 项目文档

├── .gitattributes        # Git 属性配置
├── .editorconfig         # 编辑器统一配置
└── README.md             # 项目说明
```

## 2. Git 三件套配置

> 这三个文件直接复制到新项目根目录即可

### 2.1 .gitignore — Git 忽略规则

**核心作用**：规定哪些文件不提交到代码仓库

**必加的几类**：
- `node_modules/` — 依赖包，体积大，由 package.json 管理
- `.env` — 环境变量，含密钥，绝对不能提交
- `*.db` — 数据库文件，含真实业务数据
- `dist/` / `build/` — 构建产物，可重新生成
- `.DS_Store` / `Thumbs.db` — 操作系统垃圾文件
- `*.log` — 日志文件

**完整模板见附录**，或直接从本项目根目录复制 `.gitignore` 文件。

### 2.2 .gitattributes — 换行符与二进制文件

**核心作用**：
- 统一所有文本文件的换行符为 **LF**（仓库内）
- 标记二进制文件（图片、字体、数据库等），避免 Git 乱改
- 避免 Windows/macOS 开发者混用导致的换行符混乱

**关键规则**：
```
* text=auto eol=lf          # 所有文本文件仓库内统一 LF
*.bat text eol=crlf         # Windows 批处理保持 CRLF
*.png binary                # 图片是二进制
*.db binary                 # 数据库是二进制
```

### 2.3 .editorconfig — 跨编辑器代码风格

**核心作用**：不依赖某个编辑器的设置，所有打开这个项目的编辑器自动统一风格

**关键配置**：

| 配置项 | 值 | 说明 |
|--------|-----|------|
| `indent_style` | `space` | 用空格缩进（不用 tab） |
| `indent_size` | `2` | 2 个空格（前端/Node 项目标准） |
| `end_of_line` | `lf` | 换行符 LF |
| `charset` | `utf-8` | UTF-8 编码 |
| `insert_final_newline` | `true` | 文件末尾加空行 |
| `trim_trailing_whitespace` | `true` | 去掉行尾空格 |

> **VS Code 需要装扩展**: EditorConfig for VS Code

## 3. 后端配置

### 3.1 技术栈

| 组件 | 选型 | 版本要求 | 说明 |
|------|------|---------|------|
| 运行时 | Node.js | >= 18.0.0 | 建议 18 LTS 或 20 LTS |
| 框架 | Express | 4.x | 最流行的 Node.js Web 框架 |
| 数据库 | better-sqlite3 | 9.x | 同步 API，性能好，直接操作文件 |
| 环境变量 | dotenv | 16.x | 从 .env 文件加载配置 |
| 跨域 | cors | 2.x | CORS 跨域处理 |
| 压缩 | compression | 1.x | Gzip/Brotli 响应压缩 |

### 3.2 .env 环境变量模板

> `.env` 文件**不提交 Git**，每个环境单独配置

```env
# ===== 服务端口 =====
PORT=3000

# ===== 环境模式 =====
NODE_ENV=development   # 生产环境改为 production

# ===== 管理后台 Token =====
# 管理接口需要在请求头带 x-admin-token: <你的token>
ADMIN_TOKEN=你的管理密码

# ===== 前端地址（CORS 白名单）=====
# 多个地址用逗号分隔，或在代码中配置数组
FRONTEND_URL=http://localhost:3001

# ===== 数据库路径（可选，默认 data/tongwuyiming.db）=====
# DB_PATH=/path/to/your/database.db

# ===== AI 相关（可选）=====
OPENAI_API_KEY=你的key
```

### 3.3 package.json 脚本

```json
{
  "scripts": {
    "start": "node server.js",          // 生产环境启动
    "dev": "node --watch server.js"     // 开发模式（自动重启，需 Node 18+）
  }
}
```

### 3.4 后端目录分层原则

```
请求进入 → 中间件层 → 路由层 → 服务层 → 数据库层
          (安全/限流)  (分发)   (业务逻辑)  (SQL操作)
```

- **Routes 路由层**：只负责接收请求、参数校验、调用服务、返回响应
- **Services 服务层**：放业务逻辑，比如数据处理、AI 调用、缓存策略
- **DB 数据库层**：只做 SQL 操作，不包含业务逻辑
- **Middleware 中间件层**：通用横切逻辑（安全、限流、日志）

## 5. 数据库配置

### 5.1 选型：为什么用 SQLite？

- **单文件**：整个数据库就是一个 .db 文件，备份/迁移/部署都极其简单
- **零配置**：不需要装数据库服务，不需要启动/停止
- **足够用**：中小网站（日访问量几万以内）性能完全够用
- **直接操作**：better-sqlite3 同步 API，写起来简单

### 5.2 设计原则（非常重要！）

| 原则 | 说明 |
|------|------|
| **文件即真相** | 数据库文件是唯一数据源，所有读写直接操作磁盘文件 |
| **不用内存数据库** | 不要用 sql.js 内存库 + 定时写回，会覆盖外部修改 |
| **中文表名列名** | 表名和列名用中文，和业务概念一致，好理解 |
| **WAL 模式** | 启用 WAL，读写并发更好 |
| **外键约束** | 启用外键，保证数据一致性 |
| **自动迁移** | 启动时检测缺失的列并自动添加，保证向前兼容 |

### 5.3 数据库层代码结构

```js
// db/index.js — 数据库入口
// 核心功能：
//   initDB()       — 初始化连接
//   prepare(sql)   — 预编译 SQL，返回 {all, get, run, exec}
//   exec(sql)      — 直接执行 SQL
//   reloadDB()     — 重新加载数据库文件
//   markDirty()    — 标记已修改，清空缓存
```

### 5.4 常见表设计参考

- **主数据表**：如词条表、岗位表，存核心内容
- **分类表**：如学科表、岗位分类表，存分类信息
- **关系表**：如图谱关系表，存两表之间的关联
- **用户提交表**：存用户贡献的内容，待审核
- **统计表**：存热度、点击量等统计数据

## 7. 服务器部署指南

### 7.1 服务器要求

- **操作系统**: Linux（推荐 Ubuntu 22.04 或 24.04）
- **配置**: 2核2G 起步（个人网站完全够用）
- **Node.js**: 18 LTS 或 20 LTS

### 7.2 部署方式选择

| 方式 | 难度 | 适合场景 |
|------|------|---------|
| **PM2 直接部署** | ⭐⭐ 简单 | 个人网站、小项目 |
| **Docker 部署** | ⭐⭐⭐ 中等 | 需要标准化部署 |
| **Nginx 反向代理** | ⭐⭐⭐ 中等 | 有域名、需要 HTTPS |

### 7.3 PM2 部署步骤（推荐新手）

```bash
# 1. 安装 PM2（全局）
npm install -g pm2

# 2. 进入后端目录
cd /path/to/project/backend

# 3. 用 PM2 启动
pm2 start server.js --name 项目名-backend

# 4. 设置开机自启
pm2 startup
pm2 save

# 5. 查看状态
pm2 list
pm2 logs 项目名-backend
```

> **注意**：不要使用 `pm2 start server.js -i max`（cluster 模式）。本项目的限流器和缓存基于内存 Map，cluster 模式下每个进程独立计数，限流会失效。如需多进程部署，需先将限流器改为 Redis 等共享存储。

### 7.4 前端部署

```bash
# 1. 构建
cd app
npm run build

# 2. 产物在 app/dist/ 目录
# 3. 将 dist/ 上传到服务器
# 4. 用 Nginx 托管静态文件
```

### 7.5 Nginx 配置模板

```nginx
server {
    listen 80;
    server_name yourdomain.com;

    # 前端静态文件
    location /twym/ {
        alias /path/to/app/dist/;
        try_files $uri $uri/ /twym/index.html;
    }

    # 后端 API 反向代理
    location /twym/api/ {
        proxy_pass http://127.0.0.1:3000/;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
    }
}
```

### 7.6 数据库备份

```bash
# 方式一：直接复制 .db 文件（SQLite 备份最简单）
cp data/tongwuyiming.db data/backups/backup_$(date +%Y%m%d).db

# 方式二：定时备份（用 crontab）
# 每天凌晨 3 点备份
0 3 * * * cp /path/to/data/tongwuyiming.db /path/to/data/backups/backup_$(date +\%Y\%m\%d).db
```

## 8. 安全清单

部署前逐项检查：

- [ ] **环境变量**: `.env` 文件已设为正确值，且不在 Git 中
- [ ] **管理 Token**: `ADMIN_TOKEN` 设为强密码，不是默认值
- [ ] **NODE_ENV**: 生产环境设为 `production`
- [ ] **CORS 白名单**: 不要用 `*`，明确列出允许的域名
- [ ] **数据库文件**: 不能通过 HTTP 直接访问到 .db 文件
- [ ] **HTTPS**: 生产环境必须开启 HTTPS
- [ ] **限流**: API 有限流保护，防止被刷（注意：仅单进程模式有效，cluster 模式需改用 Redis 限流）
- [ ] **SQL 注入**: 所有 SQL 用参数化查询（? 占位符），不要字符串拼接
- [ ] **备份**: 数据库有定期备份机制
- [ ] **日志**: 错误日志有记录，方便排查问题
