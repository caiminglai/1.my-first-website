# Outline 实时协作架构参考

> 来源项目：Outline (开源知识库/Wiki)
> 参考文件：`server/services/collaboration.ts`、`server/policies/cancan.ts`、`shared/editor/lib/Extension.ts`

---

## 1. 双WebSocket架构（Yjs协作 + Socket.IO事件）

Outline 采用双 WebSocket 架构，将**文档协作**和**应用事件**分离到不同的通信通道：

### 1.1 协作通道（Hocuspocus + Yjs）
```
路径: /collaboration/:documentId
协议: 原生 WebSocket (ws库)
用途: 文档内容的实时 CRDT 同步
```

核心配置：
```typescript
const wss = new WebSocket.Server({
  noServer: true,                          // 手动处理 HTTP upgrade
  maxPayload: DocumentValidation.maxStateLength, // 限制载荷大小
});
```

**关键设计：**
- `noServer: true`：不自动绑定 HTTP 服务器，通过 `server.on('upgrade')` 手动路由
- 文档 ID 从 URL 路径解析：`/collaboration/{documentId}`
- 最大载荷限制防止恶意大文档
- 错误处理区分 `Max payload size exceeded`（警告）和其他错误（错误日志）

### 1.2 事件通道（Socket.IO）
```
路径: 独立 Socket.IO 命名空间
用途: 应用级事件通知（在线状态、光标位置、评论等）
```

**双通道分离的优势：**
- 协作通道专注高频的文档同步，不受应用事件干扰
- 事件通道处理低频应用通知，可独立扩缩容
- 协作通道有严格的载荷限制，事件通道更灵活

## 2. CRDT协作模式（Hocuspocus扩展链）

Outline 使用 Hocuspocus 作为 Yjs CRDT 的服务器端实现，通过**扩展链（Extension Chain）**模式组织功能：

### 2.1 扩展链架构

```typescript
const hocuspocus = Server.configure({
  debounce: 3000,       // 防抖：3秒内的变更合并保存
  timeout: 30000,       // 超时：30秒无活动断开
  maxDebounce: 10000,   // 最大防抖间隔
  extensions: [
    // 扩展按顺序执行，形成处理链
    new Redis({ ... }),              // 1. 多实例同步（可选）
    new Throttle({ ... }),           // 2. 速率限制
    new ConnectionLimitExtension(),  // 3. 连接数限制
    new EditorVersionExtension(),    // 4. 编辑器版本兼容
    new AuthenticationExtension(),   // 5. 身份认证
    new PersistenceExtension(),      // 6. 持久化到数据库
    new APIUpdateExtension(),        // 7. API 层同步
    new ViewsExtension(),            // 8. 视图计数等
    new LoggerExtension(),           // 9. 日志记录
    new MetricsExtension(),          // 10. 性能指标
  ],
});
```

### 2.2 各扩展职责

| 扩展 | 职责 | 执行时机 |
|------|------|----------|
| Redis | 多服务器实例间的文档状态同步 | 每次变更 |
| Throttle | 防止单用户过度请求（可配置封禁时间） | 每次请求 |
| ConnectionLimitExtension | 限制单文档最大连接数 | 连接建立 |
| EditorVersionExtension | 处理不同编辑器版本的兼容性 | 连接建立 |
| AuthenticationExtension | 验证用户身份和文档访问权限 | 连接建立 |
| PersistenceExtension | 将 CRDT 状态持久化到数据库 | 防抖后保存 |
| APIUpdateExtension | 同步更新 API 层的文档元数据 | 保存后 |
| ViewsExtension | 更新文档浏览次数等统计 | 连接时 |
| LoggerExtension | 记录协作事件日志 | 全生命周期 |
| MetricsExtension | 收集性能指标（延迟、连接数等） | 全生命周期 |

### 2.3 扩展执行顺序的重要性

扩展链是**有序的管道**，每个扩展可以在请求处理的不同阶段介入：
- **前置扩展**（Redis, Throttle）：在业务逻辑前过滤
- **核心扩展**（Auth, Persistence）：处理核心业务
- **后置扩展**（Logger, Metrics）：在业务逻辑后收集信息

## 3. 基于能力的权限系统（CanCan策略）

Outline 实现了源自 Ruby cancan 库的能力权限模型：

### 3.1 核心概念

```typescript
type Ability = {
  model: Constructor;      // 执行者类型（如 User）
  action: string;          // 动作（如 "read", "update", "delete"）
  target: Constructor;     // 目标类型（如 Document）
  condition?: Condition;   // 条件函数（可选）
};
```

### 3.2 权限定义（allow）

```typescript
canCan.allow(User, ["read", "download"], Document, (user, doc) => {
  // 条件函数：返回 boolean 或 string[]（允许的字段列表）
  return user.teamId === doc.teamId;
});
```

**特点：**
- 支持多动作批量定义：`["read", "download"]`
- 支持多目标批量定义：`[Document, Collection]`
- 条件可以是函数或对象匹配
- 条件函数可返回 `boolean`（是否允许）或 `string[]`（允许的字段列表，用于部分更新）

### 3.3 权限检查（can）

```typescript
const canRead = canCan.can(user, "read", document);
```

**检查流程：**
1. 查找所有匹配的 ability（model + action + target）
2. 执行每个 ability 的 condition 函数
3. 任一 condition 返回 true 即允许
4. 特殊处理：membership 相关的条件有去重逻辑

### 3.4 权限策略的注册模式

Outline 的策略文件集中定义所有权限规则：
```
server/policies/
├── cancan.ts          # CanCan 引擎
├── document.ts        # 文档权限策略
├── collection.ts      # 集合权限策略
├── user.ts            # 用户权限策略
└── ...
```

每个策略文件调用 `canCan.allow()` 注册规则，形成声明式的权限配置。

## 4. 扩展式编辑器架构

Outline 的编辑器基于 ProseMirror，采用扩展（Extension）模式实现功能模块化：

### 4.1 Extension 基类

```typescript
export default class Extension<TOptions extends object = object> {
  options: TOptions;
  editor: Editor;

  // 核心生命周期方法
  get type() { return "extension"; }
  get name() { return ""; }
  get plugins(): Plugin[] { return []; }
  get rulePlugins(): PluginSimple[] { return []; }
  get defaultOptions(): Partial<TOptions> { return {}; }

  // 功能扩展点
  get allowInReadOnly(): boolean { return false; }
  get focusAfterExecution(): boolean { return true; }
  widget(_props: WidgetProps): React.ReactElement | undefined { return undefined; }
  keys(_options: { ... }): Record<string, Command> { return {}; }
}
```

### 4.2 扩展点设计

每个 Extension 可以提供：

| 扩展点 | 类型 | 用途 |
|--------|------|------|
| `plugins` | ProseMirror Plugin[] | 编辑器行为插件 |
| `rulePlugins` | markdown-it PluginSimple[] | Markdown 解析规则 |
| `keys` | 快捷键绑定 | 键盘命令映射 |
| `widget` | React 组件 | UI 组件（工具栏、菜单等） |
| `allowInReadOnly` | boolean | 只读模式下是否激活 |
| `options` | 泛型配置 | 可定制的扩展参数 |

### 4.3 扩展注册流程

```
Extension 实例化
    ↓
bindEditor(editor)  // 绑定编辑器实例
    ↓
合并 options (defaultOptions + 用户配置)
    ↓
注册 plugins → ProseMirror 插件系统
注册 rulePlugins → markdown-it 解析链
注册 keys → 快捷键系统
注册 widget → React Widget 渲染层
```

### 4.4 扩展类型层次

```
Extension (基础扩展)
├── Node (节点扩展 - 定义文档结构元素)
├── Mark (标记扩展 - 定义行内格式)
└── Extension (纯行为扩展 - 如快捷键、工具栏)
```

## 5. MobX Store 模式

Outline 前端使用 MobX 进行状态管理，与编辑器扩展系统深度集成：

### 5.1 Store 与 Widget 的集成

Extension 的 `widget()` 方法返回的 React 组件会自动观察 MobX store：
```typescript
// Extension 基类注释说明：
// "all widgets are observed automatically, so you can use observable values"
widget(_props: WidgetProps): React.ReactElement | undefined {
  return undefined;
}
```

### 5.2 典型 Store 结构

```
stores/
├── AuthStore.ts       # 用户认证状态
├── DocumentsStore.ts  # 文档集合状态
├── UIStore.ts         # UI 状态（主题、侧边栏等）
├── PoliciesStore.ts   # 权限策略缓存
└── ...
```

### 5.3 Store 与协作的交互

- Store 中的文档状态通过 Hocuspocus 同步
- CRDT 变更触发 Store 更新
- Store 更新通过 MobX observer 自动反映到 UI
- Widget 组件自动响应 Store 变化

## 6. 对本项目（同物异名）的适用建议

### 6.1 双通道架构的借鉴

如果"同物异名"项目需要实时协作功能（如多人协同编辑词条、共同构建知识图谱），可以参考双通道设计：

| 通道 | 用途 | 技术方案 |
|------|------|----------|
| 协作通道 | 词条内容的实时协同编辑 | Yjs CRDT + Hocuspocus |
| 事件通道 | 在线用户列表、编辑通知、评论 | Socket.IO |

**适用场景：**
- 多位专家同时编辑同一词条
- 实时显示其他用户的编辑位置
- 协同标注同物异名关系

### 6.2 扩展链模式的借鉴

知识图谱构建流程可以采用类似的扩展链模式：

```typescript
const pipeline = configurePipeline({
  extensions: [
    new DataSourceExtension(),      // 数据源接入
    new ParseExtension(),           // 词条解析
    new DeduplicationExtension(),   // 去重处理
    new LLMAnalysisExtension(),     // LLM 语义分析
    new RelationExtractExtension(), // 关系提取
    new ValidationExtension(),      // 质量校验
    new PersistenceExtension(),     // 持久化
    new MetricsExtension(),         // 指标收集
  ],
});
```

**优势：**
- 每个扩展独立开发、测试、替换
- 扩展顺序可配置，适应不同场景
- 新增功能只需添加新扩展，不修改现有代码

### 6.3 CanCan权限系统的借鉴

如果项目涉及多用户、多角色的知识贡献：

```typescript
// 权限定义示例
canCan.allow(Expert, ["create", "edit"], NameVariant, (expert, variant) => {
  return expert.domainAreas.includes(variant.domain);
});

canCan.allow(Reviewer, ["approve", "reject"], NameVariant, (reviewer, variant) => {
  return variant.submittedBy !== reviewer.id; // 不能审批自己提交的
});

canCan.allow(Admin, "manage", All, () => true);
```

**适用场景：**
- 领域专家只能编辑自己擅长领域的词条
- 审核者不能审批自己提交的内容
- 管理员拥有全部权限

### 6.4 扩展式编辑器架构的借鉴

如果项目有前端编辑器需求（词条编辑、图谱可视化配置），Extension 模式非常值得借鉴：

```typescript
// 同物异名专用编辑器扩展
class AliasHighlightExtension extends Extension {
  get name() { return "alias-highlight"; }
  get plugins() {
    return [
      // 高亮显示已知的同物异名
      createAliasHighlightPlugin(),
    ];
  }
  widget({ readOnly }) {
    return readOnly ? <AliasSidebar /> : undefined;
  }
}

class SourceReferenceExtension extends Extension {
  get name() { return "source-reference"; }
  // 文献引用的 Node 类型
  get plugins() { return [sourceReferencePlugin()]; }
}
```

### 6.5 MobX Store 模式的借鉴

前端状态管理建议：

```
stores/
├── EntityStore.ts        # 实体（事物）状态
├── NameVariantStore.ts   # 名称变体状态
├── RelationStore.ts      # 同物异名关系状态
├── SourceStore.ts        # 文献来源状态
├── AuthStore.ts          # 用户认证状态
└── UIStore.ts            # UI 状态
```

- 每个 Store 管理一个领域的数据
- Store 之间通过引用关联（如 NameVariant 关联到 Entity）
- UI 组件自动响应 Store 变化
- 协作变更通过 CRDT 同步到 Store

## 总结

Outline 的核心架构思想：
1. **双通道分离**：高频协作数据和低频应用事件走不同通道，互不干扰
2. **扩展链模式**：功能通过可插拔的扩展组织，顺序可配置，职责单一
3. **声明式权限**：CanCan 模式将权限规则集中定义，条件函数灵活表达
4. **扩展式编辑器**：Editor 功能通过 Extension 模块化，支持多种扩展点
5. **响应式状态**：MobX Store 与编辑器 Widget 自动联动，数据驱动 UI

这些模式可以迁移到"同物异名"项目中，特别是在需要多人协作构建知识图谱的场景下，双通道架构和扩展链模式能提供良好的可扩展性。
