# UI 设计规范参考 — "同物异名"项目

> 基于 beautiful-ui 设计体系提炼的实操规范。每个页面开发完成后，对照本文档逐项检查。

---

## 一、六大设计原则

### 1. 层次（Hierarchy）

- 让用户一眼看出页面的**主要任务**是什么。
- 优先使用字号对比、间距分组、对齐方式和色彩对比来建立层次，而非堆砌装饰元素。
- 内容排列顺序本身就是层次：最重要的放最前面。

**检查要点：** 页面第一眼看到的内容，是否就是用户最需要的？

### 2. 排版（Typography）

- 选择与产品气质匹配的字体风格（严肃/活泼/技术感）。
- 使用**可见的字号阶梯**（如 12 / 14 / 16 / 20 / 24 / 32px），避免随意取值。
- 在信息密度高的工具页面中，避免使用过大的标题字体。

**检查要点：** 仅看字号和粗细，能否区分标题、正文、辅助信息？

### 3. 色彩（Color）

- 色彩用于传递**含义、优先级和品牌感**。
- 避免单一色调的页面（缺乏重点）和随意渐变（无功能含义的渐变）。
- 语义色保持一致：成功=绿、警告=黄、错误=红、信息=蓝。

**检查要点：** 去掉颜色后，信息的优先级是否仍然清晰？

### 4. 间距（Spacing）

- 使用**可重复的节奏**（如 4px 基准：4 / 8 / 12 / 16 / 24 / 32 / 48px）。
- 增大间距是为了**区分分组**，不是为了让页面显得空旷。
- 相关元素间距小，不相关元素间距大。

**检查要点：** 同一组内容的元素是否靠得更近？不同组之间是否有明显间隔？

### 5. 表面（Surfaces）

- 让背景色和卡片来解释页面结构。
- 卡片适用于：重复项列表、模态框、独立工具区域。
- **不要**把每个区块都放进卡片——只有需要视觉隔离的内容才用卡片。

**检查要点：** 页面上是否每个区域都被卡片包裹了？如果是，考虑去掉不必要的卡片。

### 6. 动效（Motion）

- 动效用于说明**连续性、反馈和状态变化**。
- 保持简短（通常 150-300ms）。
- 必须尊重 `prefers-reduced-motion` 设置。

**检查要点：** 动画是否帮助用户理解了变化？关掉动画后功能是否仍然正常？

## 二、shadcn + Tailwind 组件规范

### 2.1 CSS 变量 Token 体系

在编写页面之前，先确认以下 token 已定义：

| Token 类别 | 变量示例 | 用途 |
|-----------|---------|------|
| 圆角 | `--radius` | 按钮、输入框、卡片、对话框、表格的统一圆角 |
| 前景色 | `--foreground` | 主要文字颜色 |
| 背景色 | `--background` | 页面底色 |
| 静音色 | `--muted` / `--muted-foreground` | 次要信息、禁用状态 |
| 边框色 | `--border` | 分割线、输入框边框 |
| 强调色 | `--accent` / `--accent-foreground` | 选中态、高亮区域 |
| 主色 | `--primary` / `--primary-foreground` | 主要操作按钮 |
| 危险色 | `--destructive` / `--destructive-foreground` | 删除、危险操作 |
| 动效时长 | `--duration-fast` / `--duration-normal` | 过渡动画时长 |

**原则：** 页面中不出现硬编码的颜色值或圆角值，全部通过 token 引用。

### 2.2 优先使用的组件

| 场景 | 推荐组件 |
|------|---------|
| 操作与 CTA | `Button` |
| 表单输入 | `Input`, `Textarea`, `Select`, `Checkbox`, `Switch` + 字段包装器 |
| 浮层交互 | `Dialog`, `Sheet`, `Popover`, `DropdownMenu` |
| 密集数据展示 | `Table`（而非卡片网格） |
| 模式切换 | `Tabs`, 分段控件 |
| 异步反馈 | `Toast` 或内联状态区域 |

### 2.3 变体模式（Variant Guidance）

当某个视觉状态重复出现时，应创建或复用变体，而非写一次性样式：

```
intent:    primary | secondary | danger | quiet
density:   compact | default | spacious
state:     loading | invalid | selected | disabled
surface:   plain | raised | inset | highlighted
```

**示例：** 按钮组件通过 `intent` 变体区分主要操作（primary）、次要操作（secondary）、危险操作（danger）和安静操作（quiet），而不是每次都写不同的 className。

### 2.4 状态覆盖

每个交互组件必须覆盖以下状态：

| 状态 | 要求 |
|------|------|
| 默认 | 清晰的视觉呈现 |
| Hover | 明确的交互暗示 |
| Focus | 可见的焦点环（focus ring） |
| Active/Pressed | 按下反馈 |
| Disabled | 视觉弱化 + 禁止交互 |
| Loading | 加载指示器 + 禁止重复提交 |
| Invalid | 红色边框 + 错误文字说明 |
| Selected | 明确的选中态 |

## 三、反模式清单 — 必须避免的做法

### 视觉与结构

| 反模式 | 为什么是问题 | 正确做法 |
|--------|------------|---------|
| 通用 SaaS 仪表盘风格，没有产品特色 | 用户感受不到产品个性 | 根据产品定位设计独特的视觉方向 |
| 随意渐变和阴影代替层次结构 | 视觉混乱，重点不突出 | 用字号、间距、对齐建立层次 |
| 默认字体，没有排版个性 | 显得廉价、不专业 | 选择与产品匹配的字体方案 |
| 所有内容都塞进卡片（卡片堆） | 卡片失去区分作用 | 仅对重复项、模态框、独立工具使用卡片 |

### 流程与状态

| 反模式 | 为什么是问题 | 正确做法 |
|--------|------------|---------|
| 只设计正常路径（Happy Path） | 真实场景充满异常和边界 | 必须设计空状态、加载态、错误态 |
| 跳过焦点、禁用、验证、空、加载、错误状态 | 用户体验断裂 | 逐状态检查并实现 |
| 所有平台强制相同界面 | 不同平台有不同的交互范式 | 根据平台特性做适配 |

### 开发流程

| 反模式 | 为什么是问题 | 正确做法 |
|--------|------------|---------|
| 不查组件库就自己造轮子 | 重复、不一致 | 先查已有组件，再决定是否新建 |
| 把灵感图或 Figma 稿当作不可质疑的真理 | 设计稿可能不考虑技术可行性 | 结合实际情况做合理调整 |

## 四、UI 验证清单 — 每个页面必须检查

### 4.1 视觉质量

- [ ] 视觉方向明确，具有产品特色（不是通用模板）
- [ ] 排版建立了清晰的层次（标题 > 正文 > 辅助信息）
- [ ] 间距合理地分组了相关内容
- [ ] 表面（卡片/背景）的使用是有意图的

### 4.2 状态覆盖

- [ ] 空状态（无数据时的展示）
- [ ] 加载状态（数据请求中）
- [ ] 错误状态（请求失败/表单验证失败）
- [ ] 成功状态（操作完成反馈）
- [ ] 禁用状态（不可操作的元素）
- [ ] 焦点状态（键盘导航时的焦点环）
- [ ] 选中状态（已选择的项）
- [ ] 破坏性操作的二次确认

### 4.3 无障碍

- [ ] 颜色对比度满足 WCAG AA 标准（正文 4.5:1，大文字 3:1）
- [ ] 文字缩放/适配已考虑
- [ ] 键盘导航路径清晰（Tab 顺序合理）
- [ ] 尊重 `prefers-reduced-motion` 设置
- [ ] 图标按钮有可访问名称（aria-label）
- [ ] 对话框/Sheet 有焦点管理
- [ ] 菜单和 Popover 支持键盘操作
- [ ] 表格有表头，选中状态可被读屏器识别
- [ ] 无效字段有可见的错误文字和 `aria-invalid`

### 4.4 平台适配

- [ ] 导航模式匹配目标平台（移动端底部导航 vs 桌面端侧边栏）
- [ ] 布局、密度和交互模式匹配设备和输入方式
- [ ] 多平台间的有意差异已记录说明
- [ ] 常见移动端宽度下无水平溢出

### 4.5 验证截图要求

每个页面需提供以下截图作为验证证据：

1. 桌面端完整页面
2. 移动端完整页面
3. 主要控件的 hover 和 focus 状态
4. 表单的 invalid 和 disabled 状态
5. 数据视图的 empty 和 loading 状态

## 五、质量评分标准

采用 5 分制评分，每个页面需达到以下标准：

### 评分维度

| 维度 | 权重 | 说明 |
|------|------|------|
| 视觉质量 | 25% | 层次清晰、排版有序、色彩有义、间距合理 |
| 状态覆盖 | 25% | 空/加载/错误/成功/禁用/焦点/选中状态完整 |
| 无障碍 | 20% | 对比度、键盘导航、读屏器支持、动效尊重 |
| 平台适配 | 15% | 布局匹配设备、交互匹配输入方式 |
| 组件复用 | 15% | 使用已有组件和 token，无重复造轮子 |

### 评分等级

| 分数 | 含义 |
|------|------|
| 5 | 卓越 — 可作为范例 |
| 4 | 良好 — 有小幅改进空间 |
| 3 | 合格 — 满足基本要求 |
| 2 | 不足 — 需要改进后才能发布 |
| 1 | 严重缺陷 — 需要重做 |

### 通过标准

- **总分平均 >= 3.0** 方可通过
- **无障碍、状态覆盖、视觉质量** 三个维度各自 >= 3
- **无关键失败项**（如完全缺少焦点状态、颜色对比严重不足等）

### 评分记录格式

```
页面：[页面名称]
日期：[评分日期]

视觉质量：  [1-5]  备注：...
状态覆盖：  [1-5]  备注：...
无障碍：    [1-5]  备注：...
平台适配：  [1-5]  备注：...
组件复用：  [1-5]  备注：...

平均分：    [x.x]
是否通过：  是/否
关键问题：  ...
改进建议：  ...
```

## 附录：快速参考卡片

### 间距基准（4px 网格）

```
4px   — 微间距（图标与文字之间）
8px   — 小间距（相关元素内）
12px  — 中间距（表单字段内）
16px  — 标准间距（组件间）
24px  — 大间距（区块内分组）
32px  — 区块间距（不同区块间）
48px  — 页面级间距（大段落/大区块间）
```

### 字号阶梯参考

```
12px  — 辅助文字、标签
14px  — 次要正文、表格内容
16px  — 标准正文
20px  — 小标题
24px  — 页面标题
32px  — 大标题/英雄区
```

### 动效时长参考

```
100ms — 微交互（按钮按压、开关切换）
150ms — 标准过渡（hover 效果、颜色变化）
200ms — 中等动画（下拉展开、折叠）
300ms — 较大动画（模态框出现、页面切换）
500ms — 复杂动画（仅用于重要场景）
```
