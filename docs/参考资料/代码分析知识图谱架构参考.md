# Understand-Anything 多Agent流水线架构参考

> 来源项目：Understand-Anything Plugin
> 参考文件：`skills/understand/SKILL.md`、`packages/core/src/types.ts`

## 1. 多Agent流水线架构（7阶段Pipeline）

Understand-Anything 采用严格的阶段化流水线设计，将复杂的代码分析任务分解为 7 个有序阶段：

| 阶段 | 名称 | 职责 |
|------|------|------|
| Phase 0 | Pre-flight | 环境检测、项目根目录解析、git worktree 重定向、插件构建检查、增量/全量决策逻辑 |
| Phase 0.5 | Ignore Configuration | 设置 `.understandignore` 文件，定义扫描排除规则 |
| Phase 1 | File Discovery | 扫描项目文件，按语言分类，生成文件清单 |
| Phase 2 | File Analysis | 分批（batch）对文件进行 LLM 分析，提取节点和关系 |
| Phase 3 | Cross-file Analysis | 跨文件关系分析，识别模块间依赖和架构模式 |
| Phase 4 | Graph Assembly | 将所有中间结果组装为完整的知识图谱 |
| Phase 5 | Enrichment | 对图谱进行渐进式丰富化：生成摘要、标签、复杂度评估 |
| Phase 6 | Review & Validation | 图谱质量审查，确定性校验 + 可选 LLM 审查 |
| Phase 7 | Cleanup | 清理临时文件，安全删除（mv 到 trash 而非 rm） |

**关键设计原则：**
- 每个阶段有明确的输入和输出边界
- 阶段间通过文件系统（JSON 中间文件）传递数据
- 支持增量更新：通过 git commit hash 判断是否需要重新分析
- 进度报告贯穿全流程：`[Phase N/7] <phase name>...`

## 2. 文件中介的Agent通信模式

Agent 之间不直接传递数据，而是通过**文件系统**作为中介：

```
Project Root/
└── .understand-anything/
    ├── config.json              # 配置持久化（语言、自动更新等）
    ├── meta.json                # 元信息（git commit hash）
    ├── knowledge-graph.json     # 最终输出的完整知识图谱
    ├── intermediate/
    │   └── assembled-graph.json # 组装阶段的中间产物
    └── tmp/                     # 临时分析文件
```

**通信模式特点：**
- **持久化**：中间结果写入磁盘，进程中断后可恢复
- **可审计**：每个阶段的输出都可独立检查
- **版本关联**：`meta.json` 记录 git commit hash，支持增量更新判断
- **子域合并**：支持多个子域知识图谱（如 `frontend-knowledge-graph.json`）合并为完整图谱

## 3. 确定性脚本 + LLM 配对策略

项目采用"确定性脚本负责结构，LLM 负责语义"的混合策略：

### 确定性脚本（Python/Node.js）
- 文件发现和分类（Phase 1）
- 子域图谱合并（`merge-subdomain-graphs.py`）
- 图谱结构校验（节点/边完整性检查）
- 临时文件清理（Phase 7）

### LLM 分析
- 文件内容理解和摘要生成（Phase 2）
- 跨文件关系推断（Phase 3）
- 图谱丰富化：描述、标签、复杂度评估（Phase 5）
- 可选的深度审查（`--review` 模式，Phase 6）

### 配对原则
- **先脚本后LLM**：能用确定性脚本完成的不使用 LLM
- **LLM 输出结构化**：LLM 生成内容必须符合 `GraphNode`/`GraphEdge` 的类型定义
- **确定性校验兜底**：LLM 输出后经过确定性脚本验证

## 4. 有界并行的 Fan-out 模式

Phase 2（文件分析）采用有界并行的 fan-out 模式：

```
Phase 2 执行流程：
┌──────────────────────────────────────────┐
│  文件列表 → 分批（batch）                  │
│  ┌─────────┐ ┌─────────┐ ┌─────────┐    │
│  │ Batch 1 │ │ Batch 2 │ │ Batch 3 │    │
│  │ (files) │ │ (files) │ │ (files) │    │
│  └────┬────┘ └────┬────┘ └────┬────┘    │
│       │           │           │          │
│       ▼           ▼           ▼          │
│  ┌─────────────────────────────────┐     │
│  │     合并所有批次结果              │     │
│  └─────────────────────────────────┘     │
└──────────────────────────────────────────┘
```

**关键约束：**
- 每批处理文件数有上限（有界）
- 进度报告：`Analyzing batch X/N (files: foo.ts, bar.ts, ...)`
- 批次间结果独立，可安全并行
- 合并阶段是确定性的脚本操作

## 5. 渐进式知识丰富化流程

知识图谱的构建是一个渐进丰富的过程：

### 5.1 节点类型体系（21种）
```typescript
// 5种代码节点
"file" | "function" | "class" | "module" | "concept"
// 8种非代码节点
"config" | "document" | "service" | "table" | "endpoint" | "pipeline" | "schema" | "resource"
// 3种领域节点
"domain" | "flow" | "step"
// 5种知识节点
"article" | "entity" | "topic" | "claim" | "source"
```

### 5.2 边类型体系（35种，8大类别）
- **结构关系**：imports, exports, contains, inherits, implements
- **行为关系**：calls, subscribes, publishes, middleware
- **数据流**：reads_from, writes_to, transforms, validates
- **依赖关系**：depends_on, tested_by, configures
- **语义关系**：related, similar_to
- **基础设施**：deploys, serves, provisions, triggers
- **Schema/Data**：migrates, documents, routes, defines_schema
- **领域/知识**：contains_flow, flow_step, cites, contradicts, builds_on 等

### 5.3 丰富化层次
1. **结构层**：节点和边的基础信息（id, type, name, filePath）
2. **语义层**：summary（摘要）、tags（标签）、complexity（复杂度）
3. **领域层**：DomainMeta（业务规则、实体、入口点）
4. **知识层**：KnowledgeMeta（wikilinks、backlinks、分类）
5. **导航层**：Layer（逻辑分组）、Tour（学习路径）

## 6. 对本项目（同物异名）的适用建议

### 6.1 借鉴流水线架构构建知识图谱

"同物异名"项目可以借鉴 Understand-Anything 的 7 阶段流水线，构建自己的知识图谱：

#### 推荐的阶段设计

| 阶段 | 职责 | 实现方式 |
|------|------|----------|
| Phase 0 | 数据源配置 | 确定要分析的文献/词典/数据库范围 |
| Phase 1 | 词条发现 | 从各数据源提取所有词条/名称变体 |
| Phase 2 | 词条分析 | LLM 分析每个词条的含义、来源、使用语境 |
| Phase 3 | 关联分析 | 跨词条识别同物异名关系（同一事物的不同名称） |
| Phase 4 | 图谱组装 | 合并为统一的知识图谱 |
| Phase 5 | 知识丰富 | 补充历史背景、地域分布、使用频率等元数据 |
| Phase 6 | 质量校验 | 确定性规则校验 + LLM 审查 |

#### 节点类型设计（适配同物异名领域）
```typescript
type NodeType =
  // 核心实体
  | "entity"        // 实际事物（如某种植物、某个地方）
  | "name_variant"  // 名称变体（同物异名中的不同名称）
  | "source"        // 文献/数据源
  // 分类与上下文
  | "domain"        // 领域（医学、植物学、地理等）
  | "region"        // 地域
  | "period"        // 历史时期
  // 知识节点
  | "article"       // 解释性文章
  | "claim"         // 学术论断
  | "topic";        // 主题分类
```

#### 边类型设计
```typescript
type EdgeType =
  // 核心关系
  | "alias_of"       // 同物异名关系（核心！）
  | "variant_of"     // 名称变体关系
  | "derived_from"   // 词源派生
  // 上下文关系
  | "used_in_region" // 地域使用关系
  | "used_in_period" // 时代使用关系
  | "cited_in"       // 文献引用
  // 语义关系
  | "related_to"     // 相关但非同物
  | "confused_with"  // 容易混淆的概念
  | "subcategory_of"; // 子类关系
```

### 6.2 借鉴文件中介通信模式

- 使用 JSON 文件作为中间产物存储，便于调试和增量更新
- 每个阶段的输出独立可查
- 支持子域图谱合并（如按领域分别构建再合并）

### 6.3 借鉴确定性+LLM混合策略

- **确定性脚本**负责：数据源解析、去重、格式校验、图谱结构验证
- **LLM**负责：语义分析、同物异名关系判断、词源推断、摘要生成
- **校验规则**：LLM 判断的"同物异名"关系需经过确定性规则校验（如检查是否有矛盾的分类标签）

### 6.4 借鉴渐进式丰富化

从最基础的同物异名关系开始，逐步丰富：
1. 基础层：实体 + 名称变体 + alias_of 关系
2. 语义层：添加摘要、标签、复杂度评估
3. 上下文层：地域分布、历史时期、使用领域
4. 知识层：文献引用、学术论断、解释性文章
5. 导航层：按领域/地域分组，生成学习路径

### 6.5 借鉴增量更新机制

- 记录数据源版本（类似 git commit hash）
- 新增数据源时只分析新增部分
- 支持子域独立更新后合并

## 总结

Understand-Anything 的核心架构思想：
1. **阶段化流水线**：复杂任务分解为有序阶段，每阶段职责明确
2. **文件中介通信**：Agent 间通过文件系统松耦合通信，支持断点恢复
3. **确定性+LLM混合**：结构操作用脚本，语义理解用 LLM，互为补充
4. **有界并行**：fan-out 模式加速分析，同时控制资源消耗
5. **渐进丰富**：从基础结构到丰富语义，层层递进构建知识

这些模式可以直接迁移到"同物异名"知识图谱的构建中，核心是将"代码分析"替换为"词条分析"，将"代码关系"替换为"同物异名关系"。
