# 管理后台 CRUD 架构参考

> 基于 sql-father 项目的管理后台架构模式，为"同物异名"项目的 admin 界面提供可复用的模板和最佳实践。

---

## 一、四层类型系统

整个类型体系分为四层，从全局到局部逐层细化，避免类型重复定义。

### 1.1 第一层：全局泛型（`global.d.ts`）

定义与业务无关的通用类型，所有模块共享。

```typescript
// 分页信息 —— 后端返回的分页结构
interface PageInfo<T> {
  current: number;
  size: number;
  total: number;
  records: T[];
}

// 分页请求 —— 前端发送的分页参数
interface PageRequest {
  current?: number;
  pageSize?: number;
  sortField?: string;
  sortOrder?: 'ascend' | 'descend';
}

// 删除请求 —— 所有删除操作统一用这个
interface DeleteRequest {
  id: number;
}

// 返回封装 —— 所有 API 响应的统一结构
interface BaseResponse<T> {
  code: number;
  data: T;
  message?: string;
}
```

**要点：**
- `PageRequest` 是所有查询请求的基类，业务查询类型通过 `extends PageRequest` 继承分页能力
- `BaseResponse<T>` 统一封装返回值，`code === 0` 表示成功
- `DeleteRequest` 全局复用，不需要每个模块单独定义

### 1.2 第二层：实体命名空间（`models/xxx.d.ts`）

每个业务实体一个 `.d.ts` 文件，用 `declare namespace` 组织类型。

```typescript
// models/dict.d.ts
declare namespace DictType {
  /** 实体（对应数据库表 + 后端 VO） */
  interface Dict {
    id: number;
    name: string;
    content: string;
    reviewStatus: number;
    reviewMessage?: string;
    userId: number;
    createTime: Date;
    updateTime: Date;
  }

  /** 创建请求（只包含用户需要填的字段） */
  interface DictAddRequest {
    name: string;
    content: string;
  }

  /** 更新请求（id 必填，其余可选） */
  interface DictUpdateRequest {
    id: number;
    name?: string;
    content?: string;
    reviewStatus?: number;
    reviewMessage?: string;
  }

  /** 查询请求（继承分页，字段均可选） */
  interface DictQueryRequest extends PageRequest {
    name?: string;
    content?: string;
    userId?: number;
    reviewStatus?: number;
  }
}
```

**命名规范：**
| 类型 | 命名模式 | 说明 |
|------|----------|------|
| 实体 | `XxxType.Xxx` | 完整数据对象，对应后端 VO |
| 创建请求 | `XxxType.XxxAddRequest` | 仅包含创建时可填写的字段 |
| 更新请求 | `XxxType.XxxUpdateRequest` | `id` 必填，其余可选 |
| 查询请求 | `XxxType.XxxQueryRequest` | 继承 `PageRequest`，所有字段可选 |

**同物异名项目适配示例：**
```typescript
declare namespace SynonymType {
  interface Synonym {
    id: number;
    standardName: string;     // 标准名称
    aliasName: string;        // 别名
    category?: string;        // 分类
    source?: string;          // 来源
    reviewStatus: number;
    userId: number;
    createTime: Date;
    updateTime: Date;
  }

  interface SynonymAddRequest {
    standardName: string;
    aliasName: string;
    category?: string;
    source?: string;
  }

  interface SynonymUpdateRequest {
    id: number;
    standardName?: string;
    aliasName?: string;
    category?: string;
    source?: string;
    reviewStatus?: number;
  }

  interface SynonymQueryRequest extends PageRequest {
    standardName?: string;
    aliasName?: string;
    category?: string;
    reviewStatus?: number;
  }
}
```

### 1.3 第三层：Service 层（`services/xxxService.ts`）

每个实体一个 Service 文件，封装所有 API 调用，类型全部引用命名空间中的定义。

```typescript
// services/dictService.ts
import { request } from '@umijs/max';

// 分页列表（管理后台最常用）
export async function listDictByPage(params: DictType.DictQueryRequest) {
  return request<BaseResponse<PageInfo<DictType.Dict>>>('/dict/list/page', {
    method: 'GET',
    params,
  });
}

// 创建
export async function addDict(params: DictType.DictAddRequest) {
  return request<BaseResponse<number>>('/dict/add', {
    method: 'POST',
    headers: { 'Content-Type': 'application/json' },
    data: params,
  });
}

// 根据 id 查询
export async function getDictById(id: number) {
  return request<BaseResponse<DictType.Dict>>('/dict/get', {
    method: 'GET',
    params: { id },
  });
}

// 更新
export async function updateDict(params: DictType.DictUpdateRequest) {
  return request<BaseResponse<boolean>>('/dict/update', {
    method: 'POST',
    headers: { 'Content-Type': 'application/json' },
    data: params,
  });
}

// 删除（复用全局 DeleteRequest）
export async function deleteDict(params: DeleteRequest) {
  return request<BaseResponse<boolean>>('/dict/delete', {
    method: 'POST',
    headers: { 'Content-Type': 'application/json' },
    data: params,
  });
}
```

**Service 层模式总结：**
- 函数命名：`listXxxByPage`、`addXxx`、`getXxxById`、`updateXxx`、`deleteXxx`
- GET 请求用 `params`，POST 请求用 `data` + `Content-Type` header
- 返回值全部用 `BaseResponse<T>` 包裹，泛型 T 是实际数据类型
- 删除接口统一使用全局 `DeleteRequest`

### 1.4 第四层：页面层（组件内部）

页面组件直接使用上述三层类型，不需要额外定义新类型。页面内部只有 UI 状态（如 `modalVisible`）用局部 `useState` 管理。

```typescript
// 页面内的状态只有 UI 状态，业务类型全部来自上面三层
const [createModalVisible, setCreateModalVisible] = useState<boolean>(false);
const [updateModalVisible, setUpdateModalVisible] = useState<boolean>(false);
const [updateData, setUpdateData] = useState<DictType.Dict>({} as DictType.Dict);
const actionRef = useRef<ActionType>();
```

## 二、CRUD 模板模式

### 2.1 columns 复用：一套列定义同时用于表格和表单

这是整个架构最核心的复用模式。`ProColumns` 定义一次，同时传给 `ProTable`（表格展示）和 `CreateModal/UpdateModal`（表单渲染）。

```typescript
const columns: ProColumns<DictType.Dict>[] = [
  {
    title: 'id',
    dataIndex: 'id',
    valueType: 'index',           // 序号列
  },
  {
    title: '名称',
    dataIndex: 'name',            // 同时出现在：表格列、搜索条件、新建/编辑表单
  },
  {
    title: '内容',
    dataIndex: 'content',
    valueType: 'textarea',        // 表格里截断显示，表单里渲染为 textarea
    fieldProps: {
      placeholder: '请输入内容',
    },
  },
  {
    title: '审核状态',
    dataIndex: 'reviewStatus',
    valueEnum: REVIEW_STATUS_ENUM, // 表格显示标签，表单显示下拉选择
  },
  {
    title: '创建时间',
    dataIndex: 'createTime',
    valueType: 'dateTime',
    sorter: true,                 // 支持排序
    hideInForm: true,             // 表单中隐藏（创建时不需要填）
    hideInSearch: true,           // 搜索中隐藏（不需要按时间搜索）
  },
  {
    title: '操作',
    dataIndex: 'option',
    valueType: 'option',          // 只在表格中渲染为操作按钮
    render: (_, record) => (/* 操作按钮 */),
  },
];
```

**关键控制字段：**
| 属性 | 作用 |
|------|------|
| `hideInForm: true` | 在 Modal 表单中不显示（如 id、创建时间） |
| `hideInSearch: true` | 在搜索栏中不显示（如创建时间、操作列） |
| `hideInTable: true` | 在表格中不显示（但表单/搜索中有） |
| `valueType` | 控制渲染方式：`text`、`textarea`、`dateTime`、`index`、`option` |
| `valueEnum` | 枚举映射，表格显示标签文字，表单显示下拉框 |
| `sorter: true` | 启用列排序 |

### 2.2 ProTable request 模式

```typescript
<ProTable<DictType.Dict>
  headerTitle="词库管理"
  actionRef={actionRef}           // 用于外部触发 reload
  rowKey="id"
  search={{
    labelWidth: 'auto',
    defaultCollapsed: false,      // 搜索栏默认展开
  }}
  toolBarRender={() => [
    <Button key="1" type="primary" onClick={() => setCreateModalVisible(true)}>
      新建
    </Button>,
  ]}
  request={async (params, sorter) => {
    const searchParams: DictType.DictQueryRequest = { ...params };
    // 处理排序
    for (const key in sorter) {
      searchParams.sortField = key;
      searchParams.sortOrder = sorter[key] as any;
    }
    const { data, code } = await listDictByPage(searchParams);
    return {
      data: data?.records || [],
      success: code === 0,
      total: data.total,
    } as any;
  }}
  columns={columns}
/>
```

**request 函数固定模式：**
1. 接收 `params`（搜索表单值）和 `sorter`（排序信息）
2. 将 sorter 转为后端的 `sortField` + `sortOrder` 参数
3. 调用 Service 层的 `listXxxByPage`
4. 返回 `{ data, success, total }` 格式

### 2.3 actionRef.reload() 刷新模式

所有增删改操作成功后，统一通过 `actionRef.current?.reload()` 刷新表格：

```typescript
const doDelete = async (dict: DictType.Dict) => {
  const hide = message.loading('正在删除');     // 1. 显示 loading
  if (!dict?.id) return;
  try {
    await deleteDict({ id: dict.id });          // 2. 执行操作
    message.success('操作成功');                 // 3. 成功提示
    actionRef.current?.reload();                // 4. 刷新表格
  } catch (e: any) {
    message.error('操作失败，' + e.message);    // 5. 失败提示
  } finally {
    hide();                                     // 6. 关闭 loading
  }
};
```

**loading 消息固定三步曲：**
1. `const hide = message.loading('正在xxx')` — 开始
2. `message.success('操作成功')` / `message.error('xxx失败')` — 结果
3. `hide()` — 在 `finally` 中关闭 loading

### 2.4 操作列渲染模式

```typescript
{
  title: '操作',
  dataIndex: 'option',
  valueType: 'option',
  render: (_, record) => (
    <Space split={<Divider type="vertical" />}>
      {/* 编辑 */}
      <Typography.Link onClick={() => {
        setUpdateData(record);
        setUpdateModalVisible(true);
      }}>
        修改
      </Typography.Link>

      {/* 条件性操作按钮 */}
      {record.reviewStatus !== 1 && (
        <Typography.Link onClick={() => updateReviewStatus(record, 1)}>
          通过
        </Typography.Link>
      )}

      {/* 危险操作需要 Popconfirm 确认 */}
      <Popconfirm
        title="您确定要删除么？"
        onConfirm={() => doDelete(record)}
        okText="确认"
        cancelText="取消"
      >
        <Typography.Link type="danger">删除</Typography.Link>
      </Popconfirm>
    </Space>
  ),
}
```

## 三、CreateModal / UpdateModal 通用组件模式

### 3.1 核心思路

利用 `ProTable` 的 `type="form"` 模式，将 `columns` 自动渲染为表单，无需手写任何表单项。

### 3.2 CreateModal 模板

```typescript
import { addDict } from '@/services/dictService';
import { ProColumns, ProTable } from '@ant-design/pro-components';
import { message, Modal } from 'antd';
import React, { PropsWithChildren } from 'react';

interface CreateModalProps {
  modalVisible: boolean;
  columns: ProColumns<DictType.Dict>[];   // 接收父组件的 columns
  onSubmit: () => void;
  onCancel: () => void;
}

const handleAdd = async (fields: DictType.Dict) => {
  const hide = message.loading('正在添加');
  try {
    await addDict({ ...fields } as DictType.DictAddRequest);
    hide();
    message.success('添加成功');
    return true;
  } catch (error) {
    hide();
    message.error('添加失败请重试！');
    return false;
  }
};

const CreateModal: React.FC<PropsWithChildren<CreateModalProps>> = (props) => {
  const { modalVisible, columns, onSubmit, onCancel } = props;

  return (
    <Modal
      destroyOnClose                // 关闭时销毁表单状态
      title="新建"
      open={modalVisible}
      onCancel={() => onCancel()}
      footer={null}                 // 不用默认 footer，ProTable 自带提交按钮
    >
      <ProTable<DictType.Dict, DictType.Dict>
        onSubmit={async (value) => {
          const success = await handleAdd(value);
          if (success) {
            onSubmit?.();            // 通知父组件刷新
          }
        }}
        rowKey="id"
        type="form"                 // 关键：以表单模式渲染
        columns={columns}           // 复用同一套 columns
      />
    </Modal>
  );
};

export default CreateModal;
```

### 3.3 UpdateModal 模板

```typescript
import { updateDict } from '@/services/dictService';
import { ProColumns, ProTable } from '@ant-design/pro-components';
import { message, Modal } from 'antd';
import React, { PropsWithChildren } from 'react';

interface UpdateModalProps {
  oldData: DictType.Dict;           // 当前行的数据，用于回填表单
  modalVisible: boolean;
  columns: ProColumns<DictType.Dict>[];
  onSubmit: () => void;
  onCancel: () => void;
}

const handleUpdate = async (fields: DictType.Dict) => {
  const hide = message.loading('正在更新');
  try {
    await updateDict(fields);
    hide();
    message.success('更新成功');
    return true;
  } catch (error) {
    hide();
    message.error('更新失败请重试！');
    return false;
  }
};

const UpdateModal: React.FC<PropsWithChildren<UpdateModalProps>> = (props) => {
  const { oldData, columns, modalVisible, onSubmit, onCancel } = props;

  return (
    <Modal
      destroyOnClose
      title="更新"
      open={modalVisible}           // 注意：旧版用 visible，新版用 open
      onCancel={() => onCancel()}
      footer={null}
    >
      <ProTable<DictType.Dict, DictType.Dict>
        onSubmit={async (values) => {
          const success = await handleUpdate({
            ...values,
            id: oldData.id,         // 关键：把 id 带回去
          });
          if (success) {
            onSubmit?.();
          }
        }}
        rowKey="id"
        type="form"
        form={{
          initialValues: oldData,   // 关键：用旧数据回填表单
        }}
        columns={columns}
      />
    </Modal>
  );
};

export default UpdateModal;
```

### 3.4 组件 Props 对比

| Prop | CreateModal | UpdateModal | 说明 |
|------|-------------|-------------|------|
| `modalVisible` | 有 | 有 | 控制弹窗显隐 |
| `columns` | 有 | 有 | 复用页面级的列定义 |
| `oldData` | 无 | 有 | 编辑时回填的原始数据 |
| `onSubmit` | 有 | 有 | 成功后回调（可用来刷新表格） |
| `onCancel` | 有 | 有 | 取消回调 |

### 3.5 父组件中的使用方式

```typescript
// 在 index.tsx 中
<CreateModal
  modalVisible={createModalVisible}
  columns={columns}
  onSubmit={() => {}}              // 可以留空，因为 handleAdd 内部已经处理了提示
  onCancel={() => setCreateModalVisible(false)}
/>
<UpdateModal
  modalVisible={updateModalVisible}
  oldData={updateData}
  columns={columns}
  onSubmit={() => {}}
  onCancel={() => setUpdateModalVisible(false)}
/>
```

> **注意：** `onSubmit` 回调目前传空函数。如果需要提交成功后刷新表格，可以在 `onSubmit` 中调用 `actionRef.current?.reload()`，或者直接在 `handleAdd/handleUpdate` 内部调用。

```typescript
        actionRef={actionRef}
        rowKey="id"
        search={{ labelWidth: 'auto', defaultCollapsed: false }}
        toolBarRender={() => [
          <Button key="1" type="primary" onClick={() => setCreateModalVisible(true)}>
            新建
          </Button>,
        ]}
        request={async (params, sorter) => {
          const searchParams: SynonymType.SynonymQueryRequest = { ...params };
          for (const key in sorter) {
            searchParams.sortField = key;
            searchParams.sortOrder = sorter[key] as any;
          }
          const { data, code } = await listSynonymByPage(searchParams);
          return {
            data: data?.records || [],
            success: code === 0,
            total: data.total,
          } as any;
        }}
        columns={columns}
      />
      <CreateModal
        modalVisible={createModalVisible}
        columns={columns}
        onSubmit={() => {}}
        onCancel={() => setCreateModalVisible(false)}
      />
      <UpdateModal
        modalVisible={updateModalVisible}
        oldData={updateData}
        columns={columns}
        onSubmit={() => {}}
        onCancel={() => setUpdateModalVisible(false)}
      />
    </PageContainer>
  );
};

export default AdminSynonymPage;
```

## 八、注意事项

1. **columns 中的 `hideInForm` 和 `hideInSearch`**：务必为每个字段设置正确的显隐控制，否则表单中会出现不该有的字段（如 id），或搜索栏出现无意义的字段。

2. **`destroyOnClose`**：Modal 必须设置此属性，否则关闭后再打开会残留上次的表单数据。

3. **`form={{ initialValues: oldData }}`**：UpdateModal 必须用这个方式回填数据，不能用 `useState` 手动控制表单值。

4. **loading 消息的 `hide()`**：必须在 `finally` 块中调用，确保无论成功失败都能关闭 loading。

5. **`actionRef.current?.reload()`**：只在删除和状态更新操作中使用（因为这些操作在页面内完成）。新增和编辑在 Modal 中完成，通过 `handleAdd/handleUpdate` 内部的 `message.success` 反馈，表格刷新可以在 `onSubmit` 回调中触发，也可以不刷新（用户下次搜索时自然更新）。

6. **类型断言 `as any`**：`request` 返回值和 `ProTable` 的 `request` 返回值需要 `as any` 断言，这是 umi request 和 ProTable 类型不完全匹配导致的，属于正常现象。
